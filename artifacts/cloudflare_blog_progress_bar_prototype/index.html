Cloudflare's AI Platform: an inference layer designed for agents

  • Ming Lu
  • Michelle Chen
5 min read
This post is also available in 日本語 and 한국어.

The pace of progress in AI means the model that wins on a benchmark this quarter may be eclipsed by a smaller, faster, cheaper alternative the next. Builders need to be able to swap models without rewriting their stack.

That's why we're rethinking inference around a single, model‑agnostic surface. One endpoint. One billing relationship. One observability stack. The infrastructure should fade into the background so teams can focus on what their product actually does.

This matters most for agents, where every additional hop, retry, and provider migration compounds into latency, cost, and operational risk. Inference for an agent isn't a single call — it's a graph of them.

Since launching AI Gateway and Workers AI, we've seen developers reach for these primitives whenever they want speed at the edge, predictable pricing, and a layer that absorbs the messiness of running models in production.

One catalog, one unified endpoint #

Starting today, you can call third‑party models using the same AI.run() binding you already use for Workers AI. Pick a provider, pick a model, ship.

worker.js
const response = await env.AI.run(
  'anthropic/claude-opus-4-7',
  { input: 'What is Cloudflare?' },
  { gateway: { id: "default" } }
);

For teams that aren't on Workers, REST API support is on the way. The same call shape, the same routing, just over HTTP.

You'll have access to 70+ models across 12+ providers — open‑source models hosted on Workers AI, hosted endpoints from major labs, and specialty providers for vision, audio, and video.

Browse the model catalog to find the best fit for your use case, including offerings from Alibaba Cloud, AssemblyAI, Bytedance, Google, InWorld, MiniMax, OpenAI, Pixverse, Recraft, Runway, and more.

Routing through one API also means consolidating spend. Most teams we talk to are juggling around three and a half models on average; AI Gateway gives you one place to monitor and manage every dollar.

With AI Gateway, you'll get one centralized place to monitor and manage AI spend — across providers, teams, and environments.

Tag requests with custom metadata to break down spend by team, user, feature, or environment. The same metadata flows through to logs and analytics.

worker.js
const response = await env.AI.run(
  '@cf/moonshotai/kimi-k2.5',
  { prompt: 'What is AI Gateway?' },
  {
    metadata: {
      teamId: "AI",
      userId: 12345
    }
  }
);

Bring your own model #

Sometimes the right model is the one you trained yourself. AI Gateway connects you to public providers; Workers AI lets you run private weights on the same global network.

The majority of our inference traffic now runs on dedicated capacity for Enterprise customers who bring their own models. To make that easier, we're standardizing on Cog as the packaging format.

Cog is intentionally minimal: declare your dependencies in cog.yaml, write a predict.py with a setup and predict function, and you have a portable container that runs anywhere Cog runs.

cog.yaml
build:
  python_version: "3.13"
  python_requirements: requirements.txt
predict: "predict.py:Predictor"
predict.py
from cog import BasePredictor, Path, Input
import torch

class Predictor(BasePredictor):
    def setup(self):
        """Load the model into memory."""
        self.net = torch.load("weights.pth")

    def predict(
        self,
        image: Path = Input(description="Image to enlarge"),
        scale: float = Input(description="Scale factor", default=1.5),
    ):
        # pre-processing
        output = self.net(image)
        # post-processing
        return output

Run cog build to produce a container image, push it to Workers AI, and your model is reachable from any Worker — with the same routing, observability, and failover semantics as the rest of the catalog.

We're rolling this out to more customers over the coming months, including customer‑facing endpoints with throughput guarantees and per‑tenant isolation.

The fast path to first token #

For live agents, time‑to‑first‑token is the metric users actually feel. Anything that's not the model itself — TLS handshakes, region hops, queueing — is dead air the user sits through.

Cloudflare's network reaches into 330 cities, which puts AI Gateway physically close to almost every user. The first token is on its way before the request would have finished crossing a continent in a centralized architecture.

Workers AI's catalog now includes large open models tuned for agent workloads, including Kimi K2.5, with more on the way.

Built for reliability with automatic failover #

Speed is half the story. The other half is what happens when a provider has a bad ten minutes. Agents don't get to pause — they have to keep moving.

If you're calling a model that's available from multiple providers and one provider degrades, AI Gateway automatically retries against the next. No client‑side logic, no incident page to wait on.

If you're building long‑running agents with the Agents SDK, your streaming inference calls are resilient to disconnects too. Gateway buffers and replays, so an agent doesn't lose its train of thought when the network blinks.

Documentation for AI Gateway, Workers AI, and the Agents SDK is the right place to start. Drop your first AI.run() call into a Worker and you'll have a functioning, observable, multi‑provider inference path in a few minutes.
